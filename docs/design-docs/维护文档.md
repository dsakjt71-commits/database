# Tyouqu.Database 维护文档

## 维护目标

`Tyouqu.Database` 的维护重点是保证数据库访问稳定、SQL 模板可控变更、线上问题可快速回滚，并且组件可以被多个服务长期复用。

## SQL 模板维护规范

### SQL ID 命名

推荐格式：

```text
系统.模块.动作
```

示例：

```text
auth.user.getByUsername
auth.user.create
file.metadata.search
novel.book.getById
novel.chapter.listByBookId
```

命名要求：

- 全小写。
- 使用点号分隔。
- 动作表达清楚。
- 不使用数据库类型作为前缀。

### SQL 文件命名

一个 SQL ID 对应一个 `.sql` 文件，文件名去掉 `.sql` 后就是 SQL ID：

```text
sql/sqlserver/auth/auth.user.getByUsername.sql
sql/sqlserver/auth/auth.app.getById.sql
sql/sqlserver/file/file.metadata.search.sql
sql/sqlserver/novel/novel.book.getById.sql
```

可以按业务模块分目录，但不要把多个 SQL ID 写进一个大文件。

### SQL 注释格式

允许普通 SQL 注释：

```sql
-- 根据用户名查询用户
select *
from users
where username = @Username;
```

禁止使用 `-- @id xxx` 声明 SQL ID。当前组件只从文件名读取 SQL ID，文件中出现 `-- @id ` 会直接报错。

## SQL 变更流程

### 开发环境

1. 修改 SQL 文件。
2. 本地自动热加载或重启服务。
3. 执行单元测试或接口测试。
4. 确认 SQL ID 没有重复。

### 测试环境

1. 提交 SQL 文件到代码仓库。
2. 部署测试环境。
3. 执行 SQL 模板加载校验。
4. 执行业务接口回归测试。
5. 检查慢 SQL 日志。

### 生产环境

推荐流程：

1. 备份当前 SQL 目录。
2. 上传新的 SQL 文件。
3. 执行 SQL 模板校验。
4. 调用手动 reload。
5. 观察接口错误率和慢 SQL 日志。
6. 如有异常，恢复旧 SQL 文件并再次 reload。

生产环境不建议直接开启文件监听自动生效。

## Reload 规则

手动 reload 必须满足：

- 新 SQL 全部解析成功。
- SQL ID 不存在非法重复。
- 必需 SQL ID 不缺失。
- 解析成功后整体替换内存模板。
- 解析失败时继续使用旧模板。

禁止出现部分 SQL 生效、部分 SQL 失败的状态。

## 回滚规范

每次生产 SQL 变更前保留备份：

```text
sql_backup/
  20260603_210000/
    common/
    sqlserver/
```

回滚步骤：

1. 停止继续发布。
2. 恢复上一版 SQL 文件。
3. 执行 SQL 模板 reload。
4. 验证核心接口。
5. 检查错误日志。

## 日志规范

组件应记录：

```text
SQL ID
数据库 Provider
执行耗时
影响行数
是否成功
异常类型
```

默认不记录：

```text
密码
Token
手机号完整值
邮箱完整值
身份证
银行卡
client_secret
```

如需调试参数，应仅在本地或测试环境开启敏感日志。

## 慢 SQL 管理

建议配置慢 SQL 阈值：

```json
{
  "Database": {
    "SlowSqlThresholdMilliseconds": 1000
  }
}
```

超过阈值记录警告日志：

```text
Slow SQL detected. SqlId=auth.user.search, ElapsedMs=1520, Provider=SqlServer
```

排查顺序：

1. 查看 SQL 是否缺少索引。
2. 查看分页是否没有排序字段。
3. 查看 where 条件是否导致全表扫描。
4. 查看参数是否过宽，例如 `%%`。
5. 查看数据库连接池是否耗尽。

## 多数据库维护规范

不同数据库 SQL 放在不同目录：

```text
sql/sqlserver/
sql/mysql/
sql/postgresql/
sql/sqlite/
```

如果 SQL 完全通用，可以放在：

```text
sql/common/
```

Provider 专用 SQL 会覆盖 common SQL。

新增数据库 Provider 时，需要维护：

- 方言实现。
- 连接工厂。
- Provider 注册方法。
- 对应 SQL 模板目录。
- 集成测试。

## 安全规范

所有 SQL 必须参数化。

禁止：

```csharp
$"select * from users where username = '{username}'"
```

允许：

```sql
select *
from users
where username = @Username;
```

排序字段、表名、列名等无法参数化的内容，必须使用白名单：

```csharp
var orderBy = allowedColumns.Contains(inputOrderBy)
    ? inputOrderBy
    : "created_at";
```

## 测试规范

组件应包含：

- SQL 模板解析测试。
- SQL ID 重复检测测试。
- SQL 缺失检测测试。
- SQL Server 查询测试。
- 事务提交测试。
- 事务回滚测试。
- 分页 SQL 生成测试。
- 异常包装测试。

业务系统应包含：

- Repository 测试。
- 核心接口回归测试。
- SQL 模板加载测试。

## 发布规范

组件版本建议遵循语义化版本：

```text
1.0.0
1.1.0
1.1.1
2.0.0
```

版本说明：

- 修复 bug：递增补丁版本。
- 新增兼容功能：递增次版本。
- 破坏性接口调整：递增主版本。

业务项目接入时建议锁定组件版本，不要自动漂移到最新版本。

## 常见问题

### SQL ID 找不到

检查：

- SQL 文件是否在 `RootPath` 下。
- SQL 文件扩展名是否为 `.sql`。
- SQL ID 是否等于文件名去掉 `.sql`。
- 当前 Provider 目录是否正确。
- 是否需要 reload。

### SQL 修改后没有生效

检查：

- 是否开启热加载。
- 生产环境是否需要手动 reload。
- 修改的是不是当前 Provider 的 SQL 文件。
- Provider 专用 SQL 是否覆盖了 common SQL。

### 本地能查，生产查不到

检查：

- 连接字符串是否指向同一个数据库。
- SQL 模板目录是否随部署复制。
- 表结构是否一致。
- 数据库账号权限是否一致。

### 切换数据库后报错

检查：

- Provider 是否配置正确。
- 是否引用对应 Provider 包。
- SQL 是否放到了正确目录。
- SQL 是否使用了原数据库专有语法。
- 分页语法是否由方言生成。
