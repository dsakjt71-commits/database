# Tyouqu.Database 组件设计文档

## 背景

当前认证中心数据库访问依赖 SQL Server 存储过程。虽然 Repository 层已经隔离了一部分业务代码，但 Repository 内部仍然写死了大量 `sp_xxx` 调用，导致系统和 SQL Server、存储过程绑定过紧。

如果后续要切换 MySQL、PostgreSQL、SQLite 或其他数据库，改动会集中爆发在 Repository、SQL 脚本、部署流程和数据库权限上。

因此需要抽象一个跨项目、跨数据库的通用数据库组件。

## 设计目标

- 数据库访问能力可被多个服务复用。
- 业务代码不直接依赖 SQL Server 专有能力。
- SQL 不写死在 C# 代码中。
- 支持外置 SQL 模板和按 Provider 区分 SQL。
- 支持事务、分页、参数化查询、日志、异常包装。
- 保留扩展空间，后续可以支持通用 CRUD、读写分离、慢 SQL 统计。

## 总体架构

```text
业务服务
  -> 业务 Repository
    -> Tyouqu.Database.Abstractions
      -> Tyouqu.Database.Dapper
        -> Provider / Dialect / SqlTemplateStore
          -> SQL Server / MySQL / PostgreSQL / SQLite
```

业务 Repository 只关心业务实体和 SQL 模板编号：

```csharp
public Task<User?> GetByUsernameAsync(string username)
{
    return _db.QuerySingleOrDefaultByIdAsync<User>(
        "auth.user.getByUsername",
        new { Username = username });
}
```

真实 SQL 放在外部文件中：

文件名：`auth.user.getByUsername.sql`

```sql
select *
from users
where username = @Username;
```

## 核心接口

### IDbExecutor

`IDbExecutor` 是业务 Repository 最常用的入口。

```csharp
public interface IDbExecutor
{
    Task<int> ExecuteAsync(
        string sql,
        object? parameters = null,
        CancellationToken cancellationToken = default);

    Task<IReadOnlyList<T>> QueryAsync<T>(
        string sql,
        object? parameters = null,
        CancellationToken cancellationToken = default);

    Task<T?> QuerySingleOrDefaultAsync<T>(
        string sql,
        object? parameters = null,
        CancellationToken cancellationToken = default);
}
```

### ISqlTemplateExecutor

`ISqlTemplateExecutor` 用于通过 SQL 模板编号执行 SQL。

```csharp
public interface ISqlTemplateExecutor
{
    Task<int> ExecuteByIdAsync(
        string sqlId,
        object? parameters = null,
        CancellationToken cancellationToken = default);

    Task<IReadOnlyList<T>> QueryByIdAsync<T>(
        string sqlId,
        object? parameters = null,
        CancellationToken cancellationToken = default);

    Task<T?> QuerySingleOrDefaultByIdAsync<T>(
        string sqlId,
        object? parameters = null,
        CancellationToken cancellationToken = default);
}
```

实际实现中可以让 `IDbExecutor` 同时实现 SQL 直传和 SQL 模板调用，也可以拆成两个接口。为了语义清晰，推荐拆分。

## SQL 模板设计

SQL 模板采用“一个 SQL ID 一个文件”的规则。SQL ID 等于文件名去掉 `.sql`，组件不会从 SQL 文件内容中读取 SQL ID。

示例：

```text
auth.user.getByUsername.sql -> auth.user.getByUsername
auth.user.checkEmailExists.sql -> auth.user.checkEmailExists
```

SQL 文件内容只写 SQL 和普通注释：

```sql
-- 根据用户名查询用户
select *
from users
where username = @Username;
```

解析规则：

- 每个 `.sql` 文件只对应一个 SQL ID。
- 文件名去掉 `.sql` 后就是 SQL ID。
- 文件中不能包含 `-- @id `，否则组件直接报错。
- 空白行保留。
- 参数使用 `@ParameterName`，由底层 Provider 适配。

## SQL 文件组织

```text
sql/
  common/
    audit.create.sql
  sqlserver/
    auth/
      auth.user.getByUsername.sql
      auth.app.getById.sql
      auth.token.create.sql
  mysql/
    auth/
      auth.user.getByUsername.sql
      auth.app.getById.sql
      auth.token.create.sql
  postgresql/
    auth/
      auth.user.getByUsername.sql
      auth.app.getById.sql
      auth.token.create.sql
  sqlite/
    auth/
      auth.user.getByUsername.sql
      auth.app.getById.sql
      auth.token.create.sql
```

加载顺序：

1. 先加载 `sql/common/**/*.sql`。
2. 再加载 `sql/{provider}/**/*.sql`。
3. 如果 SQL ID 重复，Provider 专用 SQL 覆盖 common SQL。

## Provider 设计

Provider 负责创建连接、注册方言、处理数据库差异。

| Provider | 连接实现 | 方言 |
| --- | --- | --- |
| SqlServer | Microsoft.Data.SqlClient | SqlServerDialect |
| MySql | MySqlConnector | MySqlDialect |
| PostgreSql | Npgsql | PostgreSqlDialect |
| Sqlite | Microsoft.Data.Sqlite | SqliteDialect |

## 热加载设计

SQL 模板支持两种加载模式：

- 启动加载：服务启动时扫描 SQL 文件。
- 手动重载：调用管理接口或内部服务触发 `ReloadAsync()`。

生产环境不建议默认开启自动文件监听，因为 SQL 改错后可能直接影响线上请求。推荐生产环境使用手动重载，并保留回滚文件。

## 与存储过程的关系

组件可以保留可选的存储过程执行接口：

```csharp
public interface IStoredProcedureExecutor
{
    Task<int> ExecuteProcedureAsync(
        string procedureName,
        object? parameters = null,
        CancellationToken cancellationToken = default);
}
```

但存储过程不是组件核心能力。核心能力应该是：

```text
外置 SQL 模板
参数化执行
多数据库 Provider
事务
分页
```
